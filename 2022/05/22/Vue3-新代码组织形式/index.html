Vue3 新代码组织形式：Composition API + <script setup>

首先，为什么要使用Compostion API 和 <Script Setup>来组织代码Vue3的代码呢，因为作者都这么说了(ﾟ∀ﾟ〃) . 下面我们先来了解一下什么是Composition API，以及使用它有什么好处。

Composition API

先举一个运用Option API的简单例子，一个累加器，并且还有一个计算属性显示累加器乘以2的结果。

使用Composition API后代码看起来很烦琐，没有Option API的写法简单好懂，但Option API的写法也有几个很严重的问题:

  • 新增功能基本上都得修改data、method 等配置，并且随着代码的不断增加， 会经常上下反复横跳，开发很痛苦。
  • 代码不好复用，组件很难抽离通用的逻辑，只能使用mixin， 但是很容易带来命名冲突的问题。

我们使用Composition API后，虽然看起来烦琐了一些，但是也带来了诸多好处：

  • 所有的API都是import引入的，用到的功能都import进来，没有用到的功能，打包的时候会被清理掉，减小包的大小。
  • 不再需要上下反复横跳，我们可以把一个功能模块的methods、data都放在一些书写，维护起来更加轻松。
  • 代码方便复用，脱离option api语法糖限制，可以把一个功能所有的method、data封装在一个独立的函数里，复用代码非常容易。
  • Composition API 新增的return 语句，可以使用<script setup>特性清除，这个我们待会再讲。

Composition API 对我们开发过程中起到了巨大的帮助。下面这个图就可以很好的说明问题: 每一个功能模块的代码颜色一样，左边是Options API， 一个功能的代码零散分布在了data 、method等配置里面，维护麻烦，而右边的Composition API就不一样了，每个功能都在一起维护。

其实还可以更进一步， 如果每个颜色块代码，我们都拆分一个函数，我们就会写出类似上面右边风格的代码，每个数据源都清晰可见，而且每个功能函数都可以在各个组件随便复用。举个小例子, 我们把antd的table的相关配置都抽取成一个common hook， 那么任何一个需要用到该配置的table 组件都可以随意复用。
